# OpenViking and Context-Mode Analysis

Reference implementations analyzed for ADR-023 v2 revision. Both validate the cognitive memory direction and contribute specific patterns adopted into the design.

## OpenViking — Context Database for AI Agents

**Source:** /home/fernando/git/references/OpenViking (Volcengine, Python + Go)

### What It Does
Unified context management for AI agents using a virtual filesystem (`viking://` protocol). Solves context fragmentation by treating memories, resources, and skills as files with tiered abstractions.

### Key Patterns Adopted

**L0/L1/L2 Tiered Abstraction:**
- L0 (~100 tokens): one-line abstract for search results and catalog
- L1 (~2K tokens): structured overview for deciding whether to load full content
- L2: full content, loaded on demand
- Generated by LLM on write, stored alongside content
- Adopted as `abstract` and `overview` columns on notes table

**Hotness Scoring:**
- `sigmoid(log1p(active_count)) × time_decay(updated_at, half_life=7d)`
- Boosts frequently accessed, recently updated contexts
- Complementary to ACT-R temporal priority (ACT-R is access-pattern-based, hotness is recency-biased)
- Adopted as a blended signal with configurable alpha (0.2)

**LLM-Assisted Deduplication:**
- Three decisions: `skip` (duplicate), `merge` (combine), `keep_both` (distinct)
- Pre-filtered by vector similarity to minimize LLM calls
- Adopted with BM25 pre-filtering instead of vector (no embedding infra yet)

**6-Category Memory Taxonomy:**
- Profile, preferences, entities, events, cases, patterns
- Adapted to Djinn's note types: added `case` and `pitfall` types for session-extracted knowledge

**Session Memory Extraction:**
- Post-session LLM reviews conversation, extracts memories by category
- Deduplicates against existing memories before storing
- Adopted as session reflection stage 2

**Convergence Detection in Retrieval:**
- BFS stops after 3 unchanged rounds for top-k results
- Prevents unbounded graph traversal
- Adopted for graph proximity signal

### What We Didn't Adopt

- **Virtual filesystem paradigm** — Djinn's markdown-on-disk + SQLite index is simpler and git-trackable
- **Intent analysis before retrieval** — generates 0-5 typed queries from session context. RRF handles signal routing without this.
- **Vector-first search** — requires embedding infrastructure. Kept as future RRF signal.
- **Heavy LLM in retrieval path** — OpenViking calls LLM for intent analysis, reranking. We keep retrieval LLM-free.
- **Python deployment** — too complex (Python 3.10+, Go toolchain for AGFS, embedding API deps, vector DB)

## context-mode — Context Optimization MCP Server

**Source:** /home/fernando/git/references/context-mode (TypeScript, ~5.4K LoC)

### What It Does
MCP server that saves 98% of context window by sandboxing tool execution and tracking session events in SQLite. Solves both context bloat (raw data in window) and session continuity (state lost on compaction).

### Key Patterns Adopted

**Priority-Tiered Context Budgeting:**
- Snapshot budget (2048 bytes) allocated by priority tier: P1 50%, P2 35%, P3-P4 15%
- Lower-priority content dropped first when budget exceeded
- Adopted for `build_context` budget parameter: seeds uncapped, L1 60% remaining, L0 40% remaining

**14-Category Event Taxonomy for Session Capture:**
- Files, tasks, rules, git, errors, environment, decisions, CWD, MCP tools, subagents, skills, role, intent, data
- Each event has type, category, priority, data, timestamp, hash
- Adopted for session reflection stage 1 event extraction

**Three-Layer Fuzzy Search:**
- Layer 1: Porter stemming (FTS5 standard) — "caching" matches "cached"
- Layer 2: Trigram substring (FTS5 trigram tokenizer) — "useEff" finds "useEffect"
- Layer 3: Levenshtein fuzzy (application code) — "kuberntes" → "kubernetes"
- Adopted as phase 11e search enhancement

**Progressive Throttling:**
- Calls 1-3: normal results. Calls 4-8: reduced results + warning. Calls 9+: blocked, redirect to batch.
- Prevents search abuse from consuming context budget
- Adopted as search degradation strategy under heavy load

**SHA256 Deduplication:**
- Hash-based dedup on event data (first 16 hex chars)
- Check last 5 events before insert
- Relevant pattern for note association deduplication

**Smart Truncation (Head + Tail):**
- When output exceeds cap: keep first 60% + last 40% of lines
- Preserves initial context AND final error messages
- Relevant for L1 overview generation — preserve beginning (purpose) and end (conclusions)

**Intent-Driven Tool Output Filtering:**
- Core insight: raw tool output should never enter the context window at full size
- When output exceeds 5KB and an intent is provided: index full output into FTS5, search for relevant sections, return only matched titles + previews + vocabulary hints
- Example: 45KB access log → 155 bytes (intent: "database connection errors" → returns only matching error lines)
- Directly applicable to memory tools: `memory_search` and `build_context` should return L0 abstracts by default, not full content
- Agent drills down via `memory_read` only for notes it actually needs

**Two-Tool Paradigm (Index-Then-Search):**
- `ctx_execute_file` for structured data: compress to summary/metrics, lose detail (95-100% savings)
- `ctx_index` + `ctx_search` for knowledge: keep exact text, return only matched sections (50-93% savings)
- Maps directly to memory tool design:
  - `memory_search` = index-then-search (returns L0 abstracts + scores, not full content)
  - `memory_read` = explicit drill-down (returns L2 full content)
  - `build_context` = budget-aware retrieval (L2 for seeds, L1 for linked, L0 for discovered)

**Batch Operations (Single-Call Multi-Query):**
- `ctx_batch_execute` combines N commands + M queries into one tool call
- Returns section inventory + search results, not raw output
- Replaces 30+ individual tool calls with one
- Applicable to memory: a `memory_batch_query` tool could combine multiple search queries + context builds into one call, reducing agent round trips

**Smart Truncation for Summary Generation:**
- 60% head + 40% tail byte budget — line-aware, UTF-8 safe
- Preserves both initial context (purpose, setup) and final conclusions (errors, results)
- Better heuristic than "first N chars" for generating L1 overviews when LLM is unavailable (fallback path)
- Useful as the non-LLM fallback: if LLM generation fails or is skipped, truncate intelligently rather than naively

**Output Compression Benchmarks (21 real scenarios):**
- 376 KB raw → 16.5 KB context (96% total savings)
- Full debugging session: 177 KB → 10.2 KB (94% more context available)
- Playwright snapshot: 56 KB → 299 B (99%)
- GitHub issues (20): 59 KB → 1.1 KB (98%)
- Git log (153 commits): 11.6 KB → 107 B (99%)
- Key takeaway: tool output compression is the single biggest lever for keeping agents under the 40% context window threshold

**Vocabulary Hints for Follow-Up Queries:**
- After indexing output, extract distinctive terms (IDF + length bonus + identifier bonus)
- Return top 40 terms as "searchable vocabulary" alongside results
- Guides agent to make precise follow-up queries instead of re-scanning
- Applicable to `memory_search`: return vocabulary of related concepts alongside results

### What We Didn't Adopt

- **Sandboxed execution** — Djinn agents already have OS-level sandboxing (ADR-013, ADR-017)
- **Hook system** — context-mode adapts to Claude Code, Gemini, Cursor hooks. Djinn owns the agent loop directly.
- **Session continuity via snapshot injection** — Djinn has session persistence in DB + compaction (ADR-018). Context-mode's approach is for external agents that don't control their own session.
- **Instruction-file routing** — context-mode teaches agents via CLAUDE.md to route calls through sandbox tools. Djinn agents call memory tools directly; routing is handled in the agent loop, not via instructions.

## Synthesis: Why LLM Calls in Memory Pipeline

Both reference implementations use LLM calls in the write path. The key insight from analyzing both:

**The cost is paid once on write, amortized across all future reads.**

- A 500-token LLM summary generated on write replaces thousands of tokens of raw content in every future `build_context`, `memory_search`, and catalog result.
- Session reflection extracts knowledge that would otherwise die with the session. One good `pattern` note prevents future agents from repeating the same discovery process.
- Deduplication prevents the note sprawl that makes search progressively worse. The cost of dedup is O(writes); the cost of not deduplicating is O(writes × reads).

The mitigation: use the cheapest adequate model (haiku-class), send minimal context (L0 abstracts, not full content), and gate LLM calls behind structural pre-filters (BM25 similarity, type compatibility) so most operations never reach the LLM.
